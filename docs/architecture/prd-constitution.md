> **Version:** 1.0.0
> **Date:** 2026-03-15
> **Author:** Beth Smith (PM)

> **Enhancement Type:** Major Feature Modification + Technology Stack Upgrade
> **Impact:** Significant (touches rules, hooks, docs, agent startup)

---

## Problem Statement

Mega Brain has two disconnected governance systems that are both incomplete:

1. **17 markdown rule files** (5,042 lines in `.claude/rules/`) -- loaded via keyword matching by `skill_router.py`. Half reference a dead 5-phase pipeline model. No enforcement beyond documentation. Each file is hand-maintained and drifts from reality.

2. **Synapse YAML engine** (`core/engine/synapse.py`, 7-layer resolver) -- only 12 rules seeded across 3 of 7 layers (L0, L1, L6). Resolves in 2.7ms but has no callers in production hooks. Built in S16 but never wired.

Neither system produces a single document that describes the full Mega Brain system. Agents load `CLAUDE.md` (which is a configuration file, not a system description) and have no awareness of the 12-step MCE pipeline, 131 ROUTING keys, 30 active hooks, or 41 registered agents. When someone modifies `core/paths.py` or adds a hook, nothing updates. The system grows but its self-awareness does not.

**Reference model:** `.aiox-core/constitution.md` demonstrates a compact governance document (170 lines, 6 principles, gate references, amendment process, versioning). We need the equivalent for Mega Brain -- generated, not hand-written.

---

## Goals

- Agents load a single, compact, accurate system description at startup
- Architecture docs regenerate automatically when key files change
- One rule system (Synapse YAML), not two (YAML + 17 markdown files)
- Valid principles from markdown rules survive the migration; stale 5-phase references do not

## Constraints

| Constraint | Impact |
|-----------|--------|
| **Brownfield system** | Pipeline MCE is live. Cannot modify `core/intelligence/pipeline/mce/*.py`. Cannot change `core/paths.py` structure (only read it). |
| **stdlib + PyYAML only** | Hook scripts run in user environments. No pip-installed packages beyond PyYAML (already distributed). No `tomli` in hooks (use `tomllib` from Python 3.11+ with graceful fallback). |
| **Context budget** | Constitution must be under 200 lines. Agents load it alongside AGENT.md + SOUL.md + MEMORY.md. Every line costs tokens. |
| **Hook performance** | PostToolUse hooks run on every tool call. Auto-update must check changed files fast (<100ms for the check) and only regenerate when necessary (<5s for full regeneration). |
| **Rule migration is destructive** | Deleting 17 `.md` files is irreversible at the working-tree level. Must be done as a single atomic commit after validation. Git history preserves the originals. |
| **Two CLAUDE.md files are protected** | Root `CLAUDE.md` and `.claude/CLAUDE.md` are Claude Code configuration -- not touchable by this project. |
| **870 uncommitted files** | Existing uncommitted changes from prior sessions. This project creates a clean branch from current state. |

| # | Decision | Options | Recommendation | Rationale |

| D-1 | **Where does constitution.md live?** | A) `docs/architecture/constitution.md` B) Root `CONSTITUTION.md` | A | Keeps governance docs together. Root is already crowded. AIOX puts theirs at `.aiox-core/constitution.md` (their equivalent of our `docs/architecture/`). |
| D-2 | **Does the auto-update hook regenerate on every PostToolUse or only on file-match?** | A) Check file path against watchlist, only regenerate on match B) Regenerate on every write | A | PostToolUse fires hundreds of times per session. Must filter first. |
| D-3 | **How to handle the tomllib dependency in hooks?** | A) Require Python 3.11+ B) Parse pyproject.toml with regex fallback C) Skip pyproject.toml reading in the hook, only in governance engine | C | Hooks should not parse TOML. The governance engine (called by the hook) already handles the fallback via `_parse_basic_toml()`. |
| D-4 | **What happens to `CLAUDE-LITE.md` after migration?** | A) Delete it B) Replace with a 10-line pointer to constitution.md C) Keep both | B | `CLAUDE-LITE.md` was designed as a lightweight startup document. A pointer achieves the same purpose without drift risk. |
| D-5 | **Migration order: rules first or constitution first?** | A) Build constitution first, then migrate rules B) Migrate rules first, then build constitution that reads them | A | Constitution generator needs to read Synapse YAML to list active rules. But the generator can also introspect the .md files during transition. Building constitution first gives us the target format to validate rule migration against. |

Avoid duplicating resolution logic. |
| D-7 | **Atomic commit or incremental?** | A) One commit: constitution + hook + rule migration B) Three commits: constitution, then hook, then migration | B | Three commits allows rollback at each stage. Rule deletion is destructive and should be its own commit. |

## Epic Structure

Single epic. Three sequential phases. Each phase is a commit boundary.

### Phase 1: Constitution Generator (FR-1, FR-2, FR-3, FR-10)

Extend `core/governance/engine.py` to produce `docs/architecture/constitution.md`. The generator reads:
- `core/intelligence/pipeline/mce/orchestrate.py` (pipeline steps)
- `agents/AGENT-INDEX.yaml` (agent count and types)
- `core/paths.py` (ROUTING key count)
- `.claude/hooks/` directory (hook count)
- `core/intelligence/rag/adaptive_router.py` (RAG pipeline modes)
- `core/engine/rules/*.yaml` (active Synapse rules)
- `pyproject.toml`, `biome.json`, `package.json` (tech stack)

Output: A compact, generated markdown file with system overview, pipeline summary, agent summary, rule summary, and governance section. Under 200 lines.

**Verification:** `python3 -m core.governance.engine generate` succeeds. Output matches expected structure. Line count under 200.

### Phase 2: Auto-Update Hook (FR-4, FR-5, FR-6, FR-12)

Create `.claude/hooks/governance_auto_update.py` (PostToolUse). Logic:
1. Read tool name and file path from stdin (PostToolUse hook contract)
2. Check if file path matches watchlist (FR-5)
3. If match: call `core.governance.engine.generate_all()`
4. If no match: exit 0 immediately
5. Timeout: 5s hard limit. On timeout, exit 1 with warning (NFR-3)

Register in `.claude/settings.json` under PostToolUse with `timeout: 30`.

**Verification:** Edit `core/paths.py`, verify `constitution.md` was regenerated. Edit an unrelated file, verify no regeneration.

### Phase 3: Rules Migration (FR-7, FR-8, FR-9, FR-11)

1. Extract valid principles from 17 `.claude/rules/*.md` files
2. Write to Synapse YAML layers:
   - L0 (constitution): security, paths, epistemic, agent integrity, formal workflow
   - L1 (global): pipeline sequencing, logging, directory contract, templates, cascading, hook standards, MCP governance
   - L6 (keywords): batch processing, agent creation, github workflow, session management, plan mode
3. Update `skill_router.py` to call `synapse.resolve_rules()` instead of reading .md files
4. Delete all 17 `.claude/rules/*.md` files
5. Replace `CLAUDE-LITE.md` content with pointer to constitution

**Verification:** `python3 -m core.engine.synapse resolve` returns 25+ rules. `skill_router.py` keyword matching still works. All 248 tests pass. `ls .claude/rules/*.md` returns empty.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Rule migration loses a valid principle | Medium | High | Diff `.claude/rules/` before and after in git. Review each rule explicitly before deletion commit. |
| Auto-update hook slows down every tool call | Low | High | File-path check is O(1) string match. Only regenerate on match. Hard 5s timeout. |
| Constitution exceeds 200 lines as system grows | Medium | Medium | Generator uses summary format (counts, not details). Details live in deep-dive docs. |
| `skill_router.py` breaks after migration | Medium | High | Phase 3 includes explicit test of keyword matching before and after. |
| Python 3.11 not available (tomllib) | Low | Low | Governance engine already has `_parse_basic_toml()` fallback. Hooks do not parse TOML directly. |
