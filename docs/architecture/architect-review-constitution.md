## 2. Risk Analysis

### Risk #1: .claude/rules/*.md are NOT loaded by skill_router.py (CRITICAL)

**Discovery:** After reading `skill_router.py` (419 lines), I confirm it reads ONLY from `SKILL-INDEX.json` (skills and sub-agents). It has zero references to `.claude/rules/*.md`. These markdown files are loaded by the Claude Code runtime as project instruction files -- they appear in the system prompt as `Contents of /path/to/file (project instructions, checked into the codebase)`.

**Impact on PRD:**

- FR-8 (delete markdown files) removes content that Claude Code currently loads into every session. This is not just a `skill_router.py` change -- it is a fundamental change to what Claude sees in its system prompt.
- FR-11 (update skill_router.py) is a no-op because skill_router.py never read the rules.
- The real migration target is `.claude/settings.json` rules configuration, not `skill_router.py`.

**Required changes:**

1. FR-11 should be rewritten: "After migration, `.claude/rules/` directory is emptied. Synapse YAML rules at `core/engine/rules/*.yaml` become the machine-readable source. A new generated summary file (or the constitution itself) is placed in `.claude/rules/` as the single rule file that Claude Code loads."
2. The phased cutover (CR-3) must ensure that a generated summary replaces the 17 markdown files BEFORE they are deleted, not after.

### Risk #2: PostToolUse hook fires on EVERY tool call (MEDIUM)

**Current state:** PostToolUse has 12 registered hooks. Adding a 13th increases cumulative hook time. Each PostToolUse event runs ALL 12 hooks sequentially.

**Measured concern:** 12 hooks * ~100ms average = ~1.2s overhead per tool call already. Adding another hook that does a string match and exits is ~50ms. Acceptable, but the cumulative load is a design concern for future additions.

**Mitigation:** The hook checks file path and exits immediately (exit 0, no output) when path does not match watchlist. This is the correct pattern. No change needed.

### Risk #3: 200-line budget is tight (LOW)

The AIOX constitution is 170 lines for 6 principles. Mega Brain has: 12-step pipeline, 107 ROUTING keys, 41 agents, 30 hooks, 25+ rules, 3 knowledge buckets, 5 RAG pipelines. Fitting all of this into 200 lines requires aggressive summarization.

**Mitigation:** Use counts and tables, not descriptions. Example: "107 ROUTING keys across 12 categories" instead of listing categories. Link to deep-dive docs for details. I estimate 160-185 lines is achievable.

### Risk #4: Timestamp idempotency (LOW)

Already noted in NFR-6. `datetime.now()` breaks strict idempotency. Low impact because the generation is triggered by file changes, not by diffing output. Fix in implementation: use a deterministic timestamp based on the newest input file's mtime.

## 3. Key Decisions Resolved

### D-1: Where does constitution.md live?

**Decision: A) `docs/architecture/constitution.md`**

Agreed. Root is crowded. Governance docs belong with architecture docs. The AIOX precedent (`.aiox-core/constitution.md`) maps to our `docs/architecture/` equivalent.

### D-2: Auto-update trigger strategy?

**Decision: A) Check file path against watchlist**

Agreed. PostToolUse fires on every tool call. Must filter. The hook reads the tool invocation from stdin, extracts the file path, and does a prefix/glob match against the watchlist. Exit 0 immediately on non-match. This is the only sane approach.

### D-3: tomllib dependency in hooks?

**Decision: C) Skip TOML parsing in hook, only in engine**

Agreed. The hook's only job is to detect file changes and call the engine. The engine already handles TOML with `_parse_basic_toml()` fallback. The hook should never import `tomllib`.

### D-4: What happens to CLAUDE-LITE.md?

**Decision: CHANGED from B to a variant.**

The PRD recommends a 10-line pointer. This is insufficient. CLAUDE-LITE.md was designed as a lightweight startup document that Claude Code loads as project instructions. A 10-line pointer has no value -- Claude cannot follow a pointer to load another file.

**Revised decision:** Delete `CLAUDE-LITE.md`. Instead, generate a compact rule summary (a "rules digest") and place it at `.claude/rules/synapse-digest.md`. This file is auto-generated by the governance engine alongside the constitution. It contains the same rules as the Synapse YAML but in a format that Claude Code can read as project instructions. Target: under 100 lines. This replaces ALL 17 markdown files with ONE generated file.

**Rationale:** This solves Risk #1. Claude Code still sees rules in its system prompt. The rules come from a generated file that cannot drift because it is regenerated whenever Synapse YAML changes. The 17 hand-maintained files are replaced by 1 auto-generated file.

### D-5: Migration order?

**Decision: A) Constitution first, then rules**

Agreed. Build the generator first. Validate it produces correct output. Then migrate rules knowing you have a validation target.

### D-6: How does skill_router.py interact with Synapse?

**Decision: CHANGED from A to D) skill_router.py does not change.**

Per Risk #1, `skill_router.py` never loaded markdown rules. It loads skills and sub-agents from SKILL-INDEX.json. This will not change. The Synapse engine serves a different purpose: it provides machine-readable rules for the governance engine and for any future enforcement hooks. These are separate systems that do not need to be merged.

However, a future hook COULD call `synapse.resolve_rules(keywords=detected_keywords)` to inject relevant rules into context -- but that is a follow-up, not this project.

### D-7: Atomic or incremental commits?

**Decision: B) Three commits -- with modification.**

Agreed on three commits. But the order changes due to Risk #1:

1. **Commit 1:** Constitution generator + auto-update hook (FR-1 through FR-6, FR-10, FR-12)
2. **Commit 2:** Rule migration to Synapse YAML + generated digest file (FR-7, FR-9, FR-11 revised)
3. **Commit 3:** Delete 17 markdown rule files (FR-8) -- only after commit 2 is validated

This sequence ensures that at no point are the rules absent from both systems.

## 7. Recommendations

1. **Add `generate_synapse_digest()` to the PRD as a formal requirement.** This is the linchpin that makes Rule #1 safe. Without it, FR-8 breaks the system.

2. **Move protocol documents to `reference/` instead of deleting.** `agent-cognition.md` (443 lines), `agent-integrity.md` (692 lines), and `epistemic-standards.md` (333 lines) are not rules -- they are agent operating protocols. Extract the 2-3 actual rules from each, then move the documents to `reference/`. The synapse-digest.md references them.

3. **Narrow the watchlist (FR-5).** The PRD lists `core/intelligence/pipeline/**/*.py` which is 28 files. Most of these are never modified. Narrow to `core/intelligence/pipeline/mce/orchestrate.py` and `core/intelligence/pipeline/mce/__init__.py`.

4. **Use deterministic timestamps.** Replace `datetime.now()` in `engine.py` with the max mtime of all input files. This makes NFR-6 (idempotency) actually true.

5. **Add a smoke test.** After Phase 3 (rule deletion), run `python3 -m core.engine.synapse resolve` and verify >= 25 rules. This is already in the PRD success criteria but should be automated as a test in `tests/python/`.
